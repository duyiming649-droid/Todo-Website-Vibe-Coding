# 错误日志 (Bug Log)

本文档记录 `todo app design` 项目中发现的 Bug，包括原因分析与修复状态。

---

## Bug #1：状态更新闭包陈旧 (Stale Closure)

**严重程度**：高  
**涉及文件**：`src/app/App.tsx`  
**修复状态**：已修复

### 问题描述

`addTodo`、`updateTodo`、`deleteTodo`、`toggleComplete` 等函数在调用 `setTodos` 时直接使用了外层闭包中的 `todos`，而非最新状态。

### 原因分析

React 的 `setState` 是异步的。当用户快速连续操作（例如快速添加多个待办、快速勾选完成）时，每次回调捕获的 `todos` 可能是同一份旧数据，导致：

- **addTodo**：第二次添加可能覆盖第一次的结果
- **updateTodo**：并发更新可能丢失
- **deleteTodo**：可能删错项或删除失败
- **toggleComplete**：快速双击可能状态回滚

### 复现步骤

1. 快速连续点击「添加待办」并提交多个待办
2. 或快速连续勾选/取消多个待办的完成状态

### 修复方案

使用 `setState` 的函数式更新，始终基于最新状态计算：

```typescript
// 修复前
setTodos([...todos, todo]);

// 修复后
setTodos(prev => [...prev, todo]);
```

## Bug #2：ID 可能重复

**严重程度**：中  
**涉及文件**：`src/app/App.tsx`  
**修复状态**：已修复

### 问题描述

`addTodo` 使用 `Date.now().toString()` 作为 `id`。在同一毫秒内添加多个待办时，会生成相同的 id，导致 React `key` 重复，可能引发渲染异常或数据混乱。

### 原因分析

`Date.now()` 返回毫秒级时间戳，高频率操作下极易碰撞。

### 修复方案

使用 `crypto.randomUUID()` 生成唯一 id：

```typescript
// 修复前
id: Date.now().toString(),

// 修复后
id: crypto.randomUUID(),
```

## Bug #3：删除无确认

**严重程度**：中（UX）  
**涉及文件**：`src/app/App.tsx`、`src/app/components/TodoItem.tsx`  
**修复状态**：已修复

### 问题描述

点击删除按钮会直接删除待办，无二次确认。用户误触会导致数据丢失。

### 原因分析

PRD / AGENTS 中要求「删除：确认后再调用 deleteTodo」，当前实现未满足。

### 修复方案

在 `TodoItem` 的删除按钮 `onClick` 中，使用 `window.confirm` 弹窗确认，用户点击「确定」后再调用 `onDelete`。

> 注：后续可改用 shadcn/ui 的 AlertDialog 以提升视觉一致性。

## Bug #4：编辑时 TodoItem 表单项未同步 props

**严重程度**：低  
**涉及文件**：`src/app/components/TodoItem.tsx`  
**修复状态**：已修复

### 问题描述

编辑表单的 `useState` 仅在组件挂载时用 `todo` 初始化。若在编辑过程中 `todo` 被父组件更新（例如从其他途径同步），表单仍显示旧数据。

### 原因分析

`useState(todo.title)` 等只在首次渲染时生效，后续 `todo` 变化不会更新本地 state。

### 修复方案

在 `todo` 或 `isEditing` 变化时，用 `useEffect` 同步表单 state：

```typescript
useEffect(() => {
  if (isEditing) {
    setEditTitle(todo.title);
    setEditDescription(todo.description || '');
    setEditCategory(todo.category);
    setEditPriority(todo.priority);
    setEditDeadline(todo.deadline || '');
  }
}, [todo, isEditing]);
```

## Bug #5：React 模块未找到 / JSX 作用域错误

**严重程度**：高（阻塞构建与类型检查）  
**涉及文件**：`src/app/App.tsx`、`package.json`  
**修复状态**：已修复

### 问题描述

1. `Cannot find module 'react' or its corresponding type declarations`  
2. `This JSX tag requires 'React' to be in scope`  
3. `Property 'key' does not exist on type 'TodoItemProps'`

### 原因分析

- **react 模块**：`package.json` 中 `react`、`react-dom` 仅作为 `peerDependencies` 且标记为 `optional`，未安装时 TypeScript 无法解析模块
- **React 作用域**：部分 TS/ESLint 配置要求 JSX 文件显式导入 `React`，否则认为 JSX 标签缺少作用域
- **key 类型错误**：上述错误导致类型推断异常，进而对 `key` 等 React 保留属性报错

### 修复方案

1. 在 `package.json` 的 `dependencies` 中添加 `react`、`react-dom`
2. 在 `devDependencies` 中添加 `@types/react`、`@types/react-dom`
3. 在 `App.tsx` 中改为 `import React, { useState, useMemo } from 'react'`

```typescript
// 修复前
import { useState, useMemo } from 'react';

// 修复后
import React, { useState, useMemo } from 'react';
```

```json
// package.json 修复
"dependencies": {
  "react": "18.3.1",
  "react-dom": "18.3.1",
  ...
},
"devDependencies": {
  "@types/react": "18.3.12",
  "@types/react-dom": "18.3.1",
  ...
}
```

## Bug #6：vite.config.ts 中 path 与 __dirname 在 ESM 下不可用

**严重程度**：中（影响构建）  
**涉及文件**：`todo app design/vite.config.ts`、`todo app design/package.json`  
**修复状态**：已修复

### 问题描述

1. `Cannot find module 'path' or its corresponding type declarations`  
2. `Cannot find name '__dirname'`

### 原因分析

- 项目使用 `"type": "module"`（ESM 模式），`__dirname` 为 CommonJS 变量，在 ESM 中不存在
- 使用 `import path from 'path'` 时，若未安装 `@types/node`，TypeScript 无法解析 Node 内置模块

### 修复方案

1. 用 `fileURLToPath` + `import.meta.url` 替代 `path.resolve(__dirname, ...)`
2. 在 `devDependencies` 中添加 `@types/node`

```typescript
// 修复前
import path from 'path'
'@': path.resolve(__dirname, './src'),

// 修复后
import { fileURLToPath } from 'node:url'
'@': fileURLToPath(new URL('./src', import.meta.url)),
```
